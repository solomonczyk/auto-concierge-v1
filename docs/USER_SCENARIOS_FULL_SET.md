# Auto‑Concierge — полный сет пользовательских сценариев

**Дата создания:** 14 марта 2026  
**Версия:** 1.0  
**Статус:** Архитектурный deliverable

---

## 1. Назначение документа

Этот документ фиксирует полный набор пользовательских сценариев для проекта Auto‑Concierge на текущем этапе архитектуры.

**Цель:**

- покрыть основные пользовательские потоки;
- выделить критичные SaaS и production-сценарии;
- отделить happy path от edge cases и failure cases;
- дать основу для backlog, QA, E2E, acceptance criteria и demo flows.

**Документ составлен по текущему известному состоянию проекта:**

- multi-tenant SaaS;
- запись клиентов в сервисный бизнес;
- публичные и административные API;
- tenant isolation;
- Telegram-бот как клиентский канал;
- webhook-driven integration;
- слои backend / frontend / admin / public;
- внешние интеграции;
- readiness/health/production hardening;
- роли superadmin / tenant admin / operator / client.

---

## 2. Границы системы

### 2.1 Что система делает

Auto‑Concierge помогает бизнесу:

- принимать заявки и записи клиентов;
- управлять услугами, слотами, магазинами/точками;
- работать через веб-интерфейс и Telegram-бота;
- синхронизировать данные с внешними системами;
- контролировать многопользовательскую SaaS-изоляцию по tenant.

### 2.2 Основные каналы

- **Public Web** — публичная запись клиента;
- **Admin / Control Plane** — внутренняя панель управления;
- **Telegram Bot** — канал общения с клиентом;
- **Webhook / API** — системные интеграции;
- **External integrations** — 1C, Alpha Auto и аналогичные провайдеры.

---

## 3. Акторы системы

### 3.1 Superadmin

Управляет платформой целиком:

- создаёт и активирует tenant;
- управляет глобальными настройками;
- подключает Telegram-ботов;
- видит системный статус;
- проводит диагностику и контроль запуска.

### 3.2 Tenant Admin

Управляет своим бизнесом внутри tenant:

- настраивает точки/shops;
- управляет услугами/services;
- настраивает Telegram-бота для своего бренда;
- контролирует запись, расписание, клиентов;
- включает/отключает фичи.

### 3.3 Operator / Manager

Работает с ежедневной операционкой:

- просматривает заявки;
- подтверждает или переносит запись;
- работает с клиентской коммуникацией;
- видит слоты и загрузку.

### 3.4 Client / End User

Конечный клиент бизнеса:

- выбирает услугу;
- выбирает точку;
- выбирает слот;
- оставляет контакты;
- получает подтверждение;
- взаимодействует через Telegram.

### 3.5 External System

Внешняя система:

- принимает синхронизацию записи;
- отвечает на webhook/provisioning/integration вызовы;
- может временно быть недоступной.

## 5. Подробные пользовательские сценарии

### Блок A. Онбординг и управление tenant

#### A1. Superadmin создаёт tenant

**Актор:** Superadmin  
**Цель:** создать нового клиента SaaS-платформы.

**Предусловия:**

- superadmin авторизован;
- система доступна;
- tenant с таким идентификатором ещё не существует.

**Основной поток:**

1. Superadmin открывает управление tenant.
2. Вводит название, slug, базовые настройки.
3. Сохраняет tenant.
4. Система валидирует уникальность.
5. Система создаёт tenant.
6. Tenant становится доступен для дальнейшей конфигурации.

**Результат:** tenant создан.

**Edge cases:** slug уже занят; невалидный формат имени/slug; БД недоступна; частично заполненные данные.

**Критичность:** высокая.

#### A2. Superadmin просматривает список tenant

**Цель:** видеть всех клиентов платформы и их состояние.

**Основной поток:** Открывает список tenant → система возвращает список → Superadmin фильтрует по статусу/поиску → открывает карточку tenant.

**Edge cases:** tenant много, нужна пагинация; один tenant повреждён по данным, но список должен открыться; tenant inactive / suspended.

#### A3. Superadmin активирует tenant

**Цель:** перевести tenant в рабочее состояние.

**Поток:** Superadmin открывает tenant → включает статус активности → система сохраняет статус → tenant доступен для публичных и внутренних операций.

**Failure cases:** tenant активирован без базовой конфигурации; без shops/services; Telegram или integrations ещё не готовы.

#### A4. Superadmin отключает tenant

**Цель:** временно остановить работу tenant.

**Ожидаемое поведение:**

- публичная запись блокируется;
- бот перестаёт принимать новые клиентские действия или возвращает controlled response;
- админские действия ограничиваются по бизнес-правилам;
- данные tenant сохраняются.

---

### Блок B. Базовая настройка бизнеса

#### B1. Tenant Admin создаёт shop

**Цель:** добавить сервисную точку.

**Поток:** Admin открывает раздел shops → заполняет название, адрес, рабочие параметры → сохраняет → система создаёт запись.

**Edge cases:** дубли названия; пустой адрес; скрытая/черновая точка; попытка создать shop вне своего tenant.

#### B2. Tenant Admin редактирует shop

**Цель:** обновить данные точки.

**Особое внимание:** изменение доступности влияет на публичные сценарии; существующие записи не теряются; архивирование точки не ломает историю.

#### B3. Tenant Admin создаёт service

**Цель:** добавить услугу, на которую можно записаться.

**Поток:** Открывает services → указывает название, длительность, доступность → привязывает к shop или tenant rules → сохраняет.

**Edge cases:** нулевая или отрицательная длительность; сервис скрыт, но используется в старых записях; сервис доступен не во всех точках.

#### B4. Tenant Admin отключает service

**Ожидаемое поведение:** новая запись блокируется; старые appointment остаются видимыми; public UI не показывает отключённую услугу.

---

### Блок C. Публичная запись клиента через web

#### C1. Клиент открывает публичную страницу записи

**Цель:** начать запись без логина.

**Поток:** Клиент открывает public URL tenant → система определяет tenant → показывает доступные shops/services → клиент начинает выбор.

**Failure cases:** tenant не найден; tenant выключен; публичная конфигурация не настроена; frontend доступен, backend недоступен.

#### C2. Клиент выбирает услугу

**Поток:** Видит список услуг → выбирает нужную → система показывает релевантные точки и/или слоты.

**Edge cases:** услуг нет; услуга временно недоступна; услуга только в одной точке; услуга скрыта по feature flag.

#### C3. Клиент выбирает shop

**Поток:** Видит точки, где доступна услуга → выбирает → система загружает слоты.

**Edge cases:** точек несколько; точка закрыта; точка удалена после открытия страницы; данные кэша устарели.

#### C4. Клиент выбирает свободный слот

**Поток:** Система показывает свободные интервалы → клиент кликает слот → система резервирует/проверяет доступность → переход к вводу данных.

**Критический риск:** гонка бронирования.

**Failure cases:** слот уже занят параллельным пользователем; слот стал недоступен между загрузкой и подтверждением; overlap protection сработал на БД уровне.

#### C5. Клиент вводит контактные данные и подтверждает запись

**Поток:** Вводит имя, телефон → подтверждает → backend создаёт appointment → возвращается подтверждение → при необходимости запускается внешняя синхронизация.

**Результат:** запись создана.

**Edge cases:** невалидный телефон; пустое обязательное поле; двойной submit; backend сохранил, sync упал; frontend получил таймаут при успешном сохранении.

#### C6. Клиент получает экран подтверждения

**Ожидаемое:** видны дата/время/услуга/точка; статус понятен; CTA на Telegram при необходимости.

#### C7. Клиент повторно отправляет форму

**Ожидаемое:** не создаётся дубль; idempotency/защита; понятный ответ пользователю.

---

### Блок D. Запись через Telegram-бот

#### D1. Tenant Admin регистрирует Telegram-бота

**Поток:** Вводит bot token/username → система создаёт запись telegram_bots → бот сохраняется в tenant.

**Edge cases:** токен неверный; бот уже привязан; tenant mismatch; не хватает обязательных полей.

#### D2. Администратор активирует Telegram-бота

**Цель:** включить рабочий webhook.

**Поток:** Открывает карточку бота → нажимает activate → система генерирует webhook secret → вызывает provisioning service → Telegram webhook устанавливается → статус обновляется.

**Критично:** единый orchestration path без дублирования webhook-логики.

**Failure cases:** Telegram API error; invalid JSON; network timeout; webhook URL неверный; secret не выставлен; частичное обновление состояния.

#### D3. Клиент впервые пишет боту

**Поток:** Клиент открывает бота → жмёт Start → Telegram отправляет webhook update → система валидирует webhook secret → update обрабатывается → бот начинает диалог.

**Edge cases:** бот не активирован; webhook secret mismatch; duplicate update; Telegram resend; tenant disabled.

#### D4. Бот помогает клиенту выбрать услугу

**Поток:** Бот показывает список → клиент выбирает → бот подгружает точки/слоты.  
**Требования:** короткий сценарий; человекопонятные ошибки; сохранение контекста сессии.

#### D5. Бот оформляет запись

**Поток:** Клиент проходит шаги → передаёт контакты → система создаёт appointment → бот отправляет подтверждение.

**Failure cases:** update пришёл дважды; Redis idempotency не сработала; запись создана, ответ не отправлен; Telegram send упал.

#### D6. Повторный update от Telegram

**Ожидаемое:** система определяет already processed; нет дубля побочного эффекта; возможен ответ already_processed.

#### D7. Вебхук без секрета или с неверным секретом

**Ожидаемое:** request отклоняется; логируется; метрики security/rejected увеличиваются; бизнес-логика не запускается.

---

### Блок E. Управление слотами и расписанием

#### E1. Система показывает доступные слоты

**Акторы:** клиент, бот, оператор.

**Поток:** Запрос слотов по shop/service/date → система считает доступность → возвращает валидные интервалы.

**Edge cases:** пустой день; сервис длиннее окон; часовой пояс; слот есть в UI, но уже недоступен.

#### E2. Защита от overlap при параллельной записи

**Поток:** Два пользователя выбирают один слот → первый подтверждает → второй подтверждает позже → DB-level protection отклоняет вторую запись → понятный ответ о недоступности.

**Критично:** production-blocking сценарий.

#### E3. Operator вручную создаёт запись в занятый интервал

**Ожидаемое:** либо запрет; либо controlled override по правилу; обязателен аудит.

---

### Блок F. Жизненный цикл appointment

#### F1. Создание appointment

**Источники:** public web, Telegram, admin.

**Эффекты:** запись с tenant_id; связь с shop/service/client; попадает в список; при необходимости — очередь интеграции.

#### F2. Оператор подтверждает appointment

**Поток:** Открывает карточку → меняет статус на CONFIRMED → сохраняется → клиенту может уйти уведомление.

#### F3. Оператор переводит в IN_PROGRESS

**Цель:** зафиксировать начало выполнения.

#### F4. Оператор завершает appointment

**Цель:** зафиксировать успешное завершение.

#### F5. Оператор отменяет appointment

**Поток:** Открывает запись → выбирает отмену → статус меняется → слот снова доступен → интеграция/уведомление при необходимости.

**Edge cases:** отмена уже завершённой; отмена чужого tenant; отмена во время сбоя внешней системы.

#### F6. Оператор переносит appointment

**Поток:** Выбирает новый слот → система проверяет конфликт → обновляет → уведомление/интеграция.

**Failure cases:** новый слот уже занят; внешняя интеграция не приняла перенос; старая дата снята, новая не записалась.

---

### Блок G. Клиентские post-booking сценарии

#### G1. Клиент хочет узнать статус записи

**Каналы:** web, bot, operator-assisted.

**Ожидаемое:** видно наличие записи; статус читаем; данные не раскрываются постороннему.

#### G2. Клиент отменяет запись

**Поток:** Открывает ссылку/бота → подтверждает отмену → система проверяет право → меняет статус.

#### G3. Клиент переносит запись

**Поток:** Запрашивает перенос → получает слоты → выбирает → система переносит.

**Edge cases:** окно переноса закрыто; запись нельзя переносить после времени; нового слота нет.

---

### Блок H. Tenant isolation и безопасность

#### H1. Tenant Admin видит только свои shops/services/appointments

**Ожидаемое:** list/get/update ограничены tenant scope; прямой доступ к чужим данным невозможен.

#### H2. Попытка доступа к чужому tenant по ID

**Ожидаемое:** 403 или 404; данные чужого tenant не раскрываются.

#### H3. Public request с неверным tenant context

**Ожидаемое:** request не смешивает данные; ошибка без утечки деталей.

#### H4. Telegram update маршрутизируется только в свой tenant

**Критично:** bot token/binding/webhook secret однозначно определяют tenant; нет cross-tenant execution.

#### H5. Оператор не обходит tenant isolation

**Ожидаемое:** tenant filter сохраняется при сложных joins; админские endpoint не ломают модель изоляции.

---

### Блок I. Внешние интеграции

#### I1. Appointment уходит во внешнюю систему после создания

**Поток:** Запись создана → определяются integration settings → integration type → отправка sync → фиксация результата.

**Результат:** данные синхронизированы или controlled failure.

#### I2. Tenant без integration settings

**Ожидаемое:** sync пропускается; не падает; success/no-op.

#### I3. Неверный integration type

**Ожидаемое:** controlled failure; диагностическая ошибка сохраняется; основной поток не разрушается.

#### I4. Внешняя система временно недоступна

**Ожидаемое:** запись сохраняется локально; интеграция ретраится или ставится в очередь; метрика ошибок растёт; нет silent failure.

#### I5. Внешняя интеграция падает на исключении

**Ожидаемое:** fail-safe; лог, метрика, статус; без падения всего request path, если допустимо.

#### I6. Redis/RQ очередь доступна

**Поток:** Job в очередь → worker забирает → sync выполняется → статус фиксируется.

#### I7. Redis/RQ недоступны

**Ожидаемое:** controlled degradation; пользовательский сценарий не рушится; оператор видит проблему с синхронизацией.

---

### Блок J. Webhook processing

#### J1. Telegram webhook обработан успешно

**Поток:** Request → валидация секрета → payload → idempotency key → обработка → ok.

#### J2. Duplicate webhook

**Ожидаемое:** duplicate распознаётся; побочные эффекты не повторяются; possible already_processed.

#### J3. Плохой JSON в webhook

**Ожидаемое:** request отклоняется; логируется; API не падает.

#### J4. Webhook endpoint доступен, зависимости не готовы

**Критично для startup-order:** сервис не принимает traffic до readiness; controlled degraded или service_healthy.

#### J5. Telegram resend после 5xx

**Ожидаемое:** повторный вызов не создаёт дублей; диалог не ломается.

---

### Блок K. Health, readiness, startup-order

#### K1. Система поднимается штатно

**Поток:** DB, Redis, API стартуют → API ждёт зависимости → readiness ok → traffic допускается.

#### K2. API стартовал раньше БД

**Ожидаемое:** readiness = not ready/degraded; traffic до готовности не допускается; service_healthy в orchestration.

#### K3. Redis недоступен, DB доступна

**Ожидаемое:** health отражает degraded; часть сценариев работает; idempotency/queue частично отключена.

#### K4. DB недоступна

**Ожидаемое:** readiness fail; запись блокируется; health даёт диагностику.

#### K5. Production deployment после релиза

**Ожидаемое:** предсказуемый startup-order; healthchecks участвуют в orchestration; webhook не принимается слишком рано.

---

### Блок L. Аутентификация, авторизация, роли

#### L1. Superadmin входит в систему

**Поток:** credentials → доступ к control plane.

#### L2. Tenant Admin входит и получает доступ только к своему tenant
#### L3. Operator не может выполнять superadmin-команды
#### L4. Неавторизованный пользователь не может открыть admin endpoints
#### L5. Неверный или истёкший токен

**Ожидаемое:** доступ отклоняется; нет утечки данных.

---

### Блок M. Наблюдаемость, аудит, метрики

#### M1. Ошибка внешней интеграции отражается в метриках

**Ожидаемое:** ошибка считается; видна в monitoring; не теряется в логах.

#### M2. Security rejection webhook отражается в логах
#### M3. Provisioning Telegram webhook оставляет audit trail
#### M4. Критические переходы статусов appointment логируются
#### M5. Оператор видит диагностический статус интеграции или provisioning

---

### Блок N. Failure / recovery сценарии

#### N1. Пользователь создал запись, ответ не дошёл

**Система должна:** избежать двойного создания при повторе; позволить безопасный repeat.

#### N2. Интеграция не ушла, запись сохранена

**Система должна:** пометить запись для sync/retry; не терять бизнес-событие.

#### N3. Webhook secret случайно сброшен

**Система должна:** перестать принимать старые webhook; позволить корректную перепривязку.

#### N4. Tenant отключён во время активной работы

**Система должна:** остановить новые действия; не портить сохранённые данные.

#### N5. Частичный сбой деплоя

**Ожидаемое:** health/readiness отражают реальность; оператор понимает, можно ли принимать клиентов.

## 7. Минимальный тестовый контур по сценариям

### 7.1 Smoke

- health/live/ready доступны;
- login работает;
- tenant list открывается;
- public page открывается;
- bot webhook отвечает корректно.

### 7.2 Core E2E

- создать tenant;
- создать shop/service;
- публично записать клиента;
- appointment сохранён;
- Telegram update → idempotent retry;
- reject без webhook secret.

### 7.3 Integration E2E

- запись без integration settings;
- запись с integration settings;
- sync failure фиксируется контролируемо.

### 7.4 Isolation E2E

- tenant A не видит tenant B;
- bot tenant A не маршрутизирует в tenant B;
- public route не смешивает данные tenant.

## 8. Acceptance criteria на уровне продукта

Система считается готовой, если:

- клиент может записаться через web без дублей и конфликтов;
- tenant isolation подтверждён на API и data level;
- Telegram webhook безопасен, идемпотентен и стабилен;
- внешняя интеграция имеет fail-safe поведение;
- production startup-order и readiness не допускают ранний приём трафика;
- оператор и администратор могут управлять жизненным циклом записи;
- ошибки видны через логи, метрики и диагностические статусы.

## 10. Вывод (техархитектор)

Наиболее важны не просто «клиент записался», а архитектурные цепочки:

1. **public booking → appointment → no overlap → safe response**
2. **telegram webhook → secret validation → idempotent processing → no duplicate side effects**
3. **appointment → external integration → fail-safe degradation**
4. **tenant scoped access → no cross-tenant leakage**
5. **deployment → readiness → safe traffic admission**

Они определяют, является ли проект production-ready SaaS, а не демо-сборкой.
