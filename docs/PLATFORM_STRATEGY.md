# Platform & Product Strategy
## Auto-Concierge System

**Дата:** 2026  
**Версия:** 1.0  
**Статус:** стратегическое решение

---

## 1. Контекст

Проект Auto-Concierge изначально разрабатывался как SaaS-решение для автосервисов (СТО) с функциональностью:

- управление записью клиентов
- контроль свободных слотов
- контроль статусов обслуживания
- операционная Kanban-доска
- уведомления
- интеграции
- мультиарендность (multi-tenant SaaS)

В процессе разработки стало очевидно, что архитектура системы решает более общий класс задач, а именно:

> **управление бронированием ресурсов, жизненным циклом заявок и операционными workflow.**

Это означает, что система потенциально применима в других индустриях.

**Примеры:**

| Индустрия | Примеры |
|-----------|---------|
| СТО | бокс, мастер, длительность ремонта |
| Медицинские клиники | врач, кабинет, повторные визиты |
| Салоны красоты | мастер, услуга, длительность |
| Аренда ресурсов | ресурс, буферы, конфликты |
| Field service | выезд, маршрут, бригада |
| Образовательные записи | курс, группа, расписание |
| Сервисные центры | очередь, приоритет, SLA |

Возникает стратегический вопрос:

> **Стоит ли сразу превращать продукт в универсальный SaaS для любых индустрий?**

---

## 2. Ключевое стратегическое решение

После анализа архитектуры, product-рисков и ресурсов принято следующее решение:

Проект развивается по модели

**Vertical Product + Platform Core**

а не по модели

**Universal SaaS с первого дня.**

---

## 3. Причины решения

### 3.1 Универсальные системы имеют высокий риск расползания

Хотя на первый взгляд системы записи похожи, между индустриями есть существенные различия:

| Индустрия | Особенности |
|-----------|-------------|
| СТО | бокс, мастер, длительность ремонта |
| Клиника | врач, кабинет, повторные визиты |
| Салон | мастер, услуга, длительность |
| Аренда | ресурс, буферы, конфликты |
| Field service | выезд, маршрут, бригада |

Около **70% функциональности совпадает**, но **30% требует разных моделей данных и логики**.

Попытка поддержать всё сразу сильно усложняет продукт.

### 3.2 Необходимость Product Truth

Перед универсализацией необходимо получить:

- реальные пользователи
- реальные рабочие процессы
- реальные ошибки эксплуатации
- реальные причины оплаты

Это возможно только при работе с **конкретной индустрией**.

### 3.3 Ограниченность ресурсов

Одновременное развитие:

1. вертикального продукта
2. универсальной платформы
3. платформенного ядра

превращает проект фактически в **три отдельных продукта**, что резко снижает вероятность успешного выхода на рынок.

---

## 4. Принятая стратегия развития

Система будет развиваться в трёх логических слоях, но как **один продукт**.

---

## 5. Архитектура уровней продукта

### 5.1 Layer 1 — Vertical Product

**Первый продукт:**

> **Auto-Concierge для СТО**

Функциональность:

- запись клиентов
- управление слотами
- контроль статусов ремонта
- Kanban-доска
- уведомления
- Telegram интеграция
- операционная панель

Это основной коммерческий продукт на текущем этапе.

**Цель:**

- подтверждение ценности
- получение реального пользовательского опыта
- выход на первых клиентов

### 5.2 Layer 2 — Platform Core

Параллельно формируется универсальное ядро системы, **независимое от конкретной индустрии**.

| Компонент | Описание |
|-----------|----------|
| **Tenant System** | multi-tenancy, роли, изоляция данных |
| **Client Management** | клиенты, контактные данные, история взаимодействий |
| **Resource Calendar** | ресурсы, доступность, временные слоты |
| **Booking Engine** | создание записи, резервирование слота, конфликт-контроль |
| **Workflow Engine** | state machine, жизненный цикл записи |
| **Notification System** | сообщения, webhook, интеграции |
| **Reliability Layer** | outbox pattern, retry logic, интеграции |
| **Audit System** | журнал событий, trace изменений |

Это и есть будущее платформенное ядро.

---

## 6. Важный архитектурный принцип

> **СТО не должно быть зашито в архитектуру.**

Нельзя допускать в базовых слоях платформы:

- `service_bay_id`
- `car_model`
- `mechanic_shift`

и подобные доменные сущности.

Все специфические сущности должны находиться в **vertical layer**.

---

## 7. Запрещённая стратегия

На текущем этапе **не допускается** запуск параллельного продукта:

> **Universal Booking SaaS**

Причины:

- преждевременная абстракция
- сложность конфигурации
- неопределённая продуктовая модель

Этот этап возможен только **после подтверждения** вертикального продукта.

---

## 8. Долгосрочная стратегия платформы

После успешного запуска первого вертикального продукта возможна экспансия.

Новые vertical продукты могут включать:

| Продукт | Фокус |
|---------|-------|
| **Clinic Concierge** | управление записью пациентов |
| **Beauty Concierge** | салоны и мастера |
| **Service Concierge** | выездные услуги |
| **Resource Booking Platform** | универсальная аренда ресурсов |

Все они должны использовать **одно platform core**.

---

## 9. Эволюция архитектуры

```
Этап 1  →  Vertical SaaS (СТО)
Этап 2  →  Platform Core стабилизируется
Этап 3  →  Запуск второго vertical продукта
Этап 4  →  Платформа становится multi-industry SaaS
```

---

## 10. Итоговое стратегическое правило

Проект развивается по формуле:

> **Vertical First — Platform Underneath — Expansion Later**

или

> **Сначала ниша → затем платформа → затем масштабирование.**

---

## 11. Практическое следствие для разработки

При разработке необходимо разделять код на:

| Слой | Содержимое |
|------|------------|
| **Vertical Layer** | специфично для СТО |
| **Platform Layer** | общая логика системы |
| **Infrastructure Layer** | auth, tenant, outbox, интеграции |

---

## 12. Ожидаемый результат стратегии

Такой подход позволяет:

- быстрее выйти на рынок
- не потерять архитектурную гибкость
- снизить риск преждевременной универсализации
- подготовить фундамент для будущей платформы

---

## Заключение

Auto-Concierge рассматривается не только как продукт для СТО, а как **будущая операционная платформа** управления сервисными workflow.

Однако развитие происходит через **вертикальный запуск**, а не через немедленную универсализацию.
